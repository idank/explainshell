# Source Extractor Failure Audit (Top 1000 CLI Import)

Date: February 23, 2026

## Context

This audit investigates why `python -m explainshell.manager --mode source` failed on a subset of a top-1000 CLI import batch.

Run context:

- Input set: 1000 selected manpages from a ranking pipeline based on Zenodo/UCI shell-command dataset context plus CommandLineFu popularity.
- Manager run: `python -m explainshell.manager --mode source --db explainshell_top1000.db /tmp/top1000_cli/manpages`
- Result: `710 extracted`, `290 failed` (failures were raised as `ExtractionError: roff parser found no options in ...`).

The source extractor currently hard-fails when `parse_options()` returns an empty list (`explainshell/source_extractor.py:27`).

## Executive Summary

Most failures are parser gaps, not truly optionless manpages.

Heuristic classification of the 290 failed pages:

- `55` are primarily section-selection misses.
- `152` are formatting/pattern misses.
- `83` are likely true no-options pages or not machine-readable by current patterns.

## Root Causes

### 1. Section-selection misses

The parser only scans sections whose names are in `_OPTION_SECTION_NAMES` (`explainshell/roff_parser.py:21`).

This misses pages where options are documented under headings not in the allowlist (for example `ALL OPTIONS`, `OPTIONS SUMMARY`, `INVOKING JQ`).

Representative examples (current parse vs. parse when scanning all sections):

- `curl.1.gz`: `0` vs `269` — options in `ALL OPTIONS`, using `.IP` (already supported format)
- `nmap.1.gz`: `0` vs `115` — options in `HOST DISCOVERY`, `TARGET SPECIFICATION`, `MISCELLANEOUS OPTIONS`, etc., using `.PP + .RS/.RE` (already supported format)
- `jq.1.gz`: `0` vs `31` — options in `INVOKING JQ`
- `ebook-convert.1.gz`: `0` vs `301` — options in `LOOK AND FEEL`, `*INPUT OPTIONS`, `*OUTPUT OPTIONS`, etc., using `.TP` (already supported format)
- `ps.1.gz`: `0` vs `59` — options in `SIMPLE PROCESS SELECTION`, `PROCESS SELECTION BY LIST`, etc., using `.TP` (already supported format)

**Verification note:** The `55` count likely understates this category. For curl, nmap, ebook-convert, and ps the option entry format is already supported by the parser — only the section name is missing. Many pages classified as "formatting misses" may actually be section-name misses where the correct section was never scanned.

### 2. Option formatting not covered by pattern recognizer

`_parse_man_options()` recognizes option entries via `.TP`, `.IP`, `.HP`, and `.PP + .RS/.RE` patterns (`explainshell/roff_parser.py:502`).

Two specific patterns are missing:

**Pattern A: `.sp` + bare flag line + `.RS/.RE` (util-linux/asciidoc style)**

Used by util-linux manpages generated by asciidoc. The structure is:
```
.sp
\fB-c\fP, \fB--command\fP \fIcommand\fP
.RS 4
Description text.
.RE
```

Representative examples: `su.1.gz`, `more.1.gz`, `logger.1.gz`, `taskset.1.gz`, `lsblk.8.gz`.

**Pattern B: `.PP` + bare flag line + plain description (go-md2man style)**

Used by Docker and other Go-based tools. The structure is:
```
.PP
\fB--debug\fP=\fItrue\fP|\fIfalse\fP
Description text.
```

No `.RS/.RE` wrapping — the description follows directly and runs until the next `.PP` or section macro.

Representative example: `docker.1.gz`.

`_parse_flag_text()` can parse both patterns' flag lines when fed directly. The issue is that `_parse_man_options()` does not recognize these as option-entry starts.

### 3. Truly optionless or near-optionless pages

Some failures appear valid:

- `break.1.gz`: explicitly indicates no parameters.
- `export.1.gz`: explicitly indicates no options.
- `true.1.gz`: no option definitions present.

There are also fish builtin pages (`cd.1.gz`, `exit.1.gz`, `help.1.gz`, `source.1.gz`) where options are only mentioned in prose (for example help flags) rather than structured option blocks.

## Data Points

From failed-set analysis:

- Failed total: `290`
- Pages with zero option sections detected: `18`
- Pages with option sections detected but still zero extracted: `272`
- Failed pages with high flag density (`flag_hits >= 5`): `191`

These numbers strongly suggest extraction logic misses, not just empty docs.

## Practical Implications

- The current failure mode (`empty -> exception`) suppresses otherwise usable pages and inflates batch failure rates.
- For large-scale imports, strict empty-as-failure is too brittle for real-world manpage diversity.
- The manager already catches `ExtractionError` at line 377 and increments a `failed` counter, so batch runs do not abort — but affected pages are silently dropped.

## Implemented Fixes

### 1. Expanded section matching

- Added contains-based matching: any section whose name contains `option` (case-insensitive) is treated as an option section.
- This covers `ALL OPTIONS`, `OPTIONS SUMMARY`, `MISCELLANEOUS OPTIONS`, `*INPUT OPTIONS`, `*OUTPUT OPTIONS`, `COMMAND LINE OPTIONS`, and similar variants without maintaining an ever-growing allowlist.

### 2. Added specific entry patterns

- **`.sp` + flag line + `.RS/.RE`**: Detects bare flag lines (lines containing `\fB` followed by `-`) after `.sp` macros, with `.RS` + description + `.RE` following. Handles util-linux/asciidoc style pages.
- **`.PP` + flag line + plain description**: When the existing `.PP + .RS/.RE` handler does not find `.RS` after the flag line, and the flag line parses successfully, collects description text until the next `.PP`, `.SH`, `.SS`, or entry macro. Handles docker/go-md2man style pages.

### 3. Added regression fixtures

Test coverage for: `curl`, `nmap`, `jq`, `ebook-convert`, `su`, `more`, `docker`, `logger`, `taskset`, `ps`.
